### 8. Parser Contract Notes

- `tool_name`: `Periphery`
- `tool_version`: `3.7.4`
- `output_format`: `json`
- `tool_output_schema_version`: `periphery-json-3.7.4`
- Input shape: top-level JSON array of declaration objects.
- Required item keys observed for parser validation: `ids`, `name`,
  `modifiers`, `location`, `attributes`, `accessibility`, `hints`, `kind`,
  `modules`.
- Unknown keys must be ignored for forward compatibility.
- Missing required keys should reject the item with a validation error; the
  parser must not infer identity without `ids`, `name`, `kind`, `location`,
  `modules`, `accessibility`, and `hints`.
- `location` is project-relative and uses 1-based line/column:
  `<path>:<line>:<column>`.
- `ids[0]` is the primary Periphery symbol id when present; retain the full
  `ids` array as evidence.
- An item is a dead-symbol candidate when `hints` contains `unused`.

## Observations

- Gate 0 could not be completed on the initial macOS 13 runner because the host
  lacks a full supported Xcode installation, and Homebrew confirms that the
  current `periphery` release requires `Xcode.app 16.4`.
- Official `Periphery 3.7.4` release assets do not unblock the host because the
  shipped macOS binary targets a newer OS/runtime than this runner provides.
- The blocker was environmental, not spec-related: the SwiftPM fixture was
  valid, and the same fixture produced raw Periphery JSON on the supported
  MacBook/Xcode host.
- Reaper remains v1 schema-ready/no-op only. Current docs continue to describe
  network upload and backend aggregation rather than a file export contract we
  could safely parse offline.

## Decision For Gate 0

- Do not start parser implementation until a real `Periphery` output capture is
  committed and signed off by `CXCodeReviewer`.
- The raw `Periphery 3.7.4` JSON fixture is now committed and ready for
  `CXCodeReviewer` Gate 0 schema sign-off.
- Reaper remains no-op/skip for v1 until a separate reviewed spec documents a
  stable offline report-file contract.
